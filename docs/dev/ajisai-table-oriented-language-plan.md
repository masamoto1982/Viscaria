# Ajisai 再企画 — Vector 指向をやめ、表計算ネイティブ言語として作りなおす

Status: 企画書 (non-canonical, 2026-07-03)

## 位置づけ（非正典・企画のみ）

- 本書は **非正典 (non-canonical)**。現行 Ajisai の意味論・ランタイム挙動・互換性方針を
  定義しない。現行の正典は `SPECIFICATION.html` のみであり、本書と正典が食い違う場合は
  正典が優先する。
- 本書は **再設計の企画** であり、実装・正典変更をまだ行わない。承認後、Phase 0 で
  正典の全面改訂ドラフトを別途提案する（§8）。
- 前回の表計算化プラン（PR #1226–#1233、PR #1234 で全面ロールバック済み）とは
  前提が異なる。前回は「既存の Vector 指向言語の上にシートを載せる」計画だったが、
  本書は「**Vector 指向そのものをやめ、言語を表計算モデルで作りなおす**」計画である。

## 0. 前回の失敗の分析 — なぜ table & cell とネスト Vector は結べなかったか

前回プラン（旧 `ajisai-spreadsheet-app-redesign-plan.md`、履歴 276c5a6 に保存）は
「シート＝辞書、セル＝ワード」の写像で、言語コアを一切変えずにホスト側へ表を置いた。
この写像自体は美しく動いたが、**値モデルが二重化した**：

1. **構造の置き場所が二つできた。** 表計算では「構造」はグリッド側にある
   （行・列・テーブルが構造）。一方 Vector 指向 Ajisai では「構造」は値の側にある
   （ネスト Vector が構造）。セルが任意のネスト Vector を保持できる限り、
   「A1:A3 という範囲」と「A1 の中の `[ [ 1 2 ] 3 ]`」という **二つの直交しない
   構造軸** が同居し、範囲演算・スピル・表示・集約のすべてで「どちらの構造を
   見るのか」という解釈分岐が発生した。
2. **範囲 ＝ Vector の写像が閉じなかった。** セル値がスカラーなら「A1:A3 → Vector」は
   自然だが、セル値が Vector だと範囲は「Vector の Vector」になり、それはユーザーが
   セル内に書いたネスト Vector と型上区別できない。MAP がグリッドの行を回るのか
   セル内の要素を回るのかが決められない。
3. **スピルの意味論が定まらない。** Vector 値のセルを隣接セルへ展開すると
   グリッド構造と値構造が相互変換されるが、ネストが絡むと往復が非可逆になる。

教訓は一つに要約できる：

> **構造の一次的な置き場所は一つでなければならない。**
> 表計算として作るなら、構造はグリッドに置き、セル値は原子に限る。

## 1. 新しい言語アイデンティティ

現行正典 §1 の宣言:

> Ajisai is an AI-first, **vector-oriented**, fractional-dataflow language.

これを次へ置き換える:

> Ajisai is an AI-first, **table-oriented**, exact-real **spreadsheet language**.
> プログラム＝ドキュメント（キャンバス上のテーブル群＋ユーザーワード辞書）。
> 評価＝再計算。構造はグリッドが持ち、セル値は原子値に限る（**Flat Cell 原則**）。

- **Flat Cell 原則**: セルが保持できるのは原子値のみ（§4.1）。列・行・テーブル・範囲
  というグリッドの構造だけが集合を表す。ネスト Vector は存在しない。
- **表計算は GUI ではなく言語モデルそのもの**。前回は「表計算はホスト機能、言語コアは
  無変更」だったが、今回は **逆転** する：テーブル・セル・依存再計算を言語コアの
  中心概念にする。前回の失敗（言語の外に表を置くと値モデルと繋がらない）の直接の帰結。
- 名前は Ajisai を続投する。アジサイの花序は「小さな花（セル）の集合が一つの花房
  （テーブル）を成す」構造であり、旧来の「Vector の花」よりむしろ新モデルに忠実である。

## 3. 廃止・置換するもの

| 廃止 | 置換先 |
| --- | --- |
| ネスト Vector（値としての可変長シーケンス） | **テーブルの列・行・範囲**。ネスト構造は「別テーブル＋キー参照」への正規化で表す（Numbers の複数テーブルと同型） |
| Record 値 | テーブルの行（ヘッダ行＝フィールド名） |
| Tensor（`DenseTensor` SoA）・VTU・暗黙並列 | **列式**（§4.3）。行ごとに独立評価なので並列性はむしろ自然に残る。SoA 最適化は将来「小分数のみの列」に対する列単位最適化として再導入可能 |
| Vector 系ワード（MAP / FILTER / FOLD / ZIP …） | MAP → 列式、FILTER → ビューテーブル（§4.3）、FOLD → 集約セル（SUM 等）、ZIP → 同一テーブルの隣接列 |
| スタックの言語アイデンティティ | 数式の**内部評価機構**としての連接（RPN）評価は残る（§4.5）が、「ユーザーが操作するスタック」「複数値が残るスタック表示」は表からは見せない。セル式の結果は常に 1 原子値 |
| CodeBlock をセル値にすること | 不可。関数は DEF で辞書に置く（セルは値、辞書はコード、の分離） |
| Child runtime / Supervisor | 本企画のスコープ外へ（将来「非同期セル」の基盤として再検討） |

旧 Ajisai コードとの**互換は目標にしない**（breaking redesign）。ただし算術・比較・論理・
QUANTIZE/CONSERVE など原子値上の語彙は同名・同意味で残し、資産 #1〜#4 の
テスト群（MC/DC 含む）を新コアの受け入れテストとして流用する。

## 4. 新コア設計

### 4.1 値モデル — セル値は原子のみ

```
CellValue := Scalar        — 厳密実数（遅延連分数、正典 §4.2 を無改造で継承）
           | Boolean       — TRUE / FALSE（Scalar と非交和、現行 §4.1 の規律を継承）
           | Text          — 不変文字列（現行 String(Arc<str>) を継承）
           | Nil(reason)   — 理由つき欠損（Bubble、診断メタデータつき）
           | Unknown(diag) — 判定未決（比較予算超過）
           | Empty         — 未入力セル（Nil とは区別する。集約は Empty を黙って飛ばし、
                             Nil は理由つきで伝播する — 表計算の慣行と Bubble Rule の両立）
```

- Vector / Record / CodeBlock / Handle は **CellValue に存在しない**。これが Flat Cell 原則。
- Empty の新設が現行との唯一の型追加。SUM が空セルで NIL にならないための、
  表計算慣行との橋渡しである（Empty は「意図的な不在」、Nil は「失敗した計算」）。

### 4.2 テーブルモデル — Numbers 型を継承

前回プランの Numbers 型（フリーキャンバス上の有限・命名テーブル）の判断は正しかったので
そのまま継承する。変わるのは「テーブル＝辞書」ではなく **テーブル＝言語コアの第一級
構造** になる点。

- ドキュメント ＝ キャンバス（テーブル配置メタデータ）＋ テーブル群 ＋ ユーザーワード辞書。
- テーブル ＝ 名前つき有限グリッド（既定 `TABLE1`）。任意でヘッダ行（列名）を持つ。
- セル参照: 同一テーブル内 `A1`、列名参照 `[@価格]`（同じ行の「価格」列）、
  テーブル越境 `TABLE2@B3`・`TABLE2@[売上]`（修飾名 `@` の構文を継承）。
- **範囲（Range）は参照であって値ではない**。`A1:A10` や `[売上]`（列全体）は
  集約語（SUM / COUNT / MAX / FOLD 系）の引数位置にのみ出現でき、セルに格納できない。
  前回の失敗点 2（範囲＝Vector の写像が閉じない）はこれで構造的に消える。

### 4.3 列式 — 旧 Vector 演算の後継

Vector 指向で MAP が担っていた「一括変換」は、**列式**（column formula）が引き継ぐ:

- 一つの列に一つの式を与えると、各行で行文脈（`[@col]` 参照）つきで評価される。
  例: 金額列 ＝ `[@単価] [@数量] *`（infix 表面形では `=[@単価]*[@数量]`）。
- 行ごとの評価は互いに独立（同一行の他列と、他テーブルのみ参照可）なので、
  行方向の並列評価が意味論を変えずに可能。旧 implicit-parallelism 路線の後継。
- FILTER は**ビューテーブル**（他テーブルへの述語つき射影として定義されるテーブル）で
  置き換える。第一段では集約語の述語引数（COUNTIF/SUMIF 相当）まで、
  ビューテーブル本体は後続フェーズ（§8 Phase 5）。

### 4.4 再計算セマンティクス — 評価＝再計算

- ドキュメントは**純粋データフロー**。セル式は参照するセル・範囲・ワードのみに依存し、
  副作用を持たない（効果はホストコマンドとして境界を渡る、という現行 Gate 原則を継承）。
- 変更時、資産 #5 の逆依存インデックスで dirty 閉包を取り、トポロジカル順に再評価する。
- **決定論**: 再計算の結果は再計算の順序・並列度に依存しない（正典に必須要件として明記）。
- 循環参照 → 該当セルは `Nil(reason=referenceCycle)`（サイクル全体が診断に載る）。
- 比較予算超過 → `Unknown`（診断つき）。無限 CF 同士の比較が止まらない問題は
  現行と同じ予算機構で「未決」に落とす。

### 4.5 数式言語 — 一つの中核、二つの表面

- **canonical 形は連接（RPN）**: `A1 B1 +`、`[@単価] [@数量] *`、`A1:A10 SUM`。
  トークナイザ・評価器・DEF・修飾子など現行資産を最大限流用でき、AI が機械的に
  読み書きしやすい（AI-first）。
- **infix 表面形**: `=A1+B1`、`=SUM(A1:A10)`。表計算ユーザーの期待に応える入力面。
  `surface_forms.rs` の伝統に従い、infix ⇄ canonical の**相互変換可能な表面層**として
  定義し、意味論は canonical 側にのみ置く。GUI の数式バーはどちらの表示も選べる。
- DEF したユーザーワードは両表面から同名で呼べる。「セル式も関数定義も同じ Ajisai」
  という前回プランの核心はここで実現される。

### 4.6 QUANTIZE と CONSERVE の昇格

- **QUANTIZE ＝ 列/セルの数値書式の実体**。既存表計算の「書式」は表示だけ丸めて値は
  浮動小数点のまま、という不整合を持つ。新 Ajisai では書式適用＝QUANTIZE 適用であり、
  丸め残差（exact residual）が診断として保存される。会計用途での差別化点。
- **CONSERVE ＝ テーブル不変量**。「明細列の合計 ＝ 合計セル」のような値保存則を
  テーブルに宣言でき、再計算のたびに検査され、破れは理由つきで表示される。

### 4.7 アーキテクチャ

```
Rust コア（新: table-core）
  ├ types/            … 継承: continued_fraction / fraction / interval（無改造）
  ├ cell_value        … 新: §4.1 の原子値（Nil/Unknown の semantic メタデータは継承）
  ├ table_store       … 新: テーブル・列・行・範囲参照
  ├ formula/          … 継承+縮小: トークナイザ・連接評価器（Vector 系語彙を除去、
  │                      行文脈と範囲集約を追加）、surface_forms（infix 表面形）
  ├ recalc/           … 継承: 依存グラフ・逆依存インデックス → dirty 閉包・トポ再計算
  └ dictionary/       … 継承: DEF・修飾名・モジュール
```

## 5. UI 設計 — Numbers の WYSIWYG、Google Sheets のシェル

方針を一文で: **キャンバスとテーブルの見た目・操作は Apple Numbers の WYSIWYG、
アプリの外殻（ファイルとシートの関係・クローム・デスクトップ/モバイル対応）は
Google スプレッドシート** を手本にする。

### 5.1 旧実装から継承する見た目の原則

ロールバックした旧実装（履歴 276c5a6、PR #1228–#1233）は言語モデルこそ廃案になったが、
ルック＆フィールの判断は所有者の意図の表明として本企画でもそのまま正とする:

1. **連続する一枚のクローム**: ヘッダ → ツールバー → 数式バー → キャンバス → フッタは
   一続きの面であり、帯同士は 1px のヘアラインで接する。帯ごとの枠線や隙間は作らない。
   **角丸はキャンバスに浮かぶテーブルカードだけ**（クローム中で唯一の丸み）。
2. **テーブルカード**: 白いカードが淡色キャンバスに浮かび、控えめな影を持つ。
   テーブル名は小さなレタースペーシングつき見出し。テーブルは内容に合わせた
   有限サイズで、行・列の追加はカード縁のアフォーダンスから行う。
   矢印キー移動でテーブルが無言に成長することはない（Numbers 流儀）。
3. **Google Sheets 型数式バー**: 名前ボックス（選択アドレスの表示、アドレス入力で
   ジャンプ）＋ イタリックの fx ラベル ＋ 等幅フォントの数式フィールド。
   セル内編集と数式バーは常時相互ミラーし、入力中はセル側にライブプレビューが出る。
4. **ツールバーは最初から完成形を置く**: 印刷・undo/redo・書式ボタンは先に
   レイアウトし、機能が実装されるたびに一つずつ有効化する。クロームは後から動かない。
5. **意味論由来のセル表示**: 数値は右揃え tabular-nums、Nil / Unknown は薄字
   イタリック、エラー・循環は赤系文字＋淡赤背景。これらの意味論表示は
   ユーザー書式より常に優先する（部分失敗の可視性を書式で塗り潰させない）。
6. **実装様式**: 外部グリッドライブラリに依存しない自前実装。モダン CSS
   （ネイティブネスト・論理プロパティ・oklch・スコープ付きカスタムプロパティ）。

### 5.2 ドキュメントモデル — ファイル＞シート＞テーブル＞セル

ファイルとシートの関係は Google スプレッドシートを踏襲する:

```
ファイル（ドキュメント） … 保存・命名・共有の単位（Google Sheets の 1 ファイル相当）
 └ シート（タブ）        … 画面下端のタブバーで切替。1 シート＝1 キャンバス
     └ テーブル          … キャンバス上に複数浮かぶ有限グリッド（Numbers 型、§4.2）
         └ セル          … 原子値（§4.1）
```

- Google Sheets の「1 シート＝1 無限グリッド」に対し、本企画は
  「1 シート＝1 キャンバス、その上に有限テーブルが複数」（Numbers 型）。
  **外殻の操作系（ファイル・タブ・切替）は Google Sheets、キャンバスの中身は
  Numbers** という分担である。
- **テーブル名はファイル内で一意**（推奨）: 参照は常に `TABLE@A1` 形で済み、
  シート修飾の段数が増えない。テーブルをシート間で移動しても参照が壊れない。
  シートは表示上のグルーピング（プレゼンテーション層）であり、言語コアが見るのは
  「ファイル＝テーブル集合＋ユーザーワード辞書」だけになる。
  （Numbers は「シート内一意＋ `シート::テーブル::セル` 修飾」だが、
  修飾を増やさない案を推す。未決 §7-6）
- ファイル操作（新規・複製・リネーム・自動保存・最近使ったファイルの一覧）は
  Google Sheets / Drive の操作感を手本にし、実体は既存 platform adapter
  （IndexedDB / Tauri）上のローカル保存から始める。
- **ユーザーワード辞書はファイルに属する**。Apps Script がファイルに紐づくのと
  同型で、「ファイル＝ブック＋関数束」という単位で持ち運べる。

### 5.3 デスクトップモード

- クローム構成は Google Sheets のまま: タイトル/メニュー帯 → ツールバー →
  数式バー → キャンバス → **下端のシートタブバー**（タブ追加ボタンつき）。
  §5.1-1 のヘアライン接続で一枚の面にする。
- キーボード流儀も Google Sheets 準拠: 矢印で移動、Enter で下・Tab で右へ確定移動、
  タイプ開始で置換編集、F2 / ダブルクリックでセル内編集、Esc で破棄、
  他所クリックで確定。範囲ドラッグ選択は編集中の式への参照挿入に使える。
- Script ビュー（DEF 編集面、§4.7）へはメニューから遷移する
  （「拡張機能 > Apps Script」に対応する動線）。

### 5.4 モバイルモード

Google Sheets の iOS / Android アプリの文法に合わせる:

- **閲覧が既定、編集へは明示的に入る**: タップ＝選択、選択セルの再タップ
  （またはダブルタップ）で編集開始。編集中はキーボード直上に数式バーが
  ドッキングする。
- 常設ツールバーは置かず、**選択コンテキストのボトムバー**（書式・クリア等）に畳む。
- シートタブは下端に維持。キャンバスはピンチズームと慣性スクロールで操作する。
  Numbers 型の有限テーブルは小画面と相性が良い（無限グリッドをさまよわない）。
- 実装は同一 DOM のレスポンシブ切替を基本とし、ポインタ種別（coarse / fine）で
  アフォーダンスを出し分ける。既存 `mobile-view-switcher` のパネル切替資産は
  Script ビュー側で温存する。

### 5.5 WYSIWYG の含意 — 「見たまま」は「厳密値の忠実な射影」

Numbers 型 WYSIWYG を掲げる以上、「見えているもの＝印刷・共有されるもの」に加え、
本企画では「見えているもの＝**厳密値の忠実な射影**」までを WYSIWYG の意味に含める:

- 表示丸めは QUANTIZE（§4.6）と連動し、丸めが起きているセルはその旨が判別できる
  （残差は診断に保存）。見た目と値の乖離を UI が隠さない。
- Nil / Unknown / 循環の意味論表示は書式で塗り潰せない（§5.1-5）。
- セル結合・書式・列幅などのプレゼンテーションは文書ストアに保存し、
  値モデル（テーブル＋辞書）には入れない。旧実装プランの
  「結合は表示スパン・参照は anchor へ透過解決・エクスポートは論理モデルから」
  という方式（旧 §3.4–3.5）を新モデルでもそのまま踏襲する。

## 6. 表計算としての差別化（なぜ作る価値があるか）

| 既存表計算の弱点 | 新 Ajisai |
| --- | --- |
| 浮動小数点（`0.1+0.2≠0.3`、金額計算の丸め誤差） | 遅延連分数による厳密実数。√ を含んでも厳密 |
| エラー値が不透明（`#DIV/0!`） | 理由つき Nil。診断メタデータが下流まで伝播 |
| 「書式」は表示だけで値と乖離 | QUANTIZE：書式＝値の量子化、残差を保存 |
| 合計の整合性は人が目視 | CONSERVE：値保存則を宣言・自動検査 |
| 式言語と拡張言語が分裂（式 vs VBA/GAS） | セル式もワード定義も同じ Ajisai |
| 決められない比較が誤答になる | UNKNOWN：未決が未決として見える |
