# Ajisai 関数型基盤ノート — LAMBDA を導入せずに純粋関数型であるために

Status: design note (non-canonical, 2026-07-04)

## 位置づけ

- 本書は **非正典**。言語意味論の正典は（新エンジンについては）`SPECIFICATION.html`
  ドラフト、出荷中ランタイムについては `SPECIFICATION.html`。矛盾する場合は正典が優先。
- 本書は「新 Ajisai を純粋関数型言語としてどう完成させるか」の設計方針と、
  それに伴う構文決定（丸括弧の解放、§4）を定める。§4 は正典ドラフトへ同時に反映する。
  §3 の関数表は **提案** であり、承認後に正典ドラフトへ昇格する。

## 1. 「偶然の関数型」から「設計された関数型」へ

「表計算ソフトは世界で最も普及している関数型プログラミング言語である」という見方は
Simon Peyton Jones らが以前から唱え、Excel の `LAMBDA`（2021、これにより式言語単体で
チューリング完全になった）が補強した。しかし歴史的経緯は逆輸入である。Excel は

- 命令的マクロ（VBA）と式言語の分裂、
- 参照透過でない関数（`NOW` `RAND` `INDIRECT`）と揮発性再計算、
- 循環参照の反復計算モード（事実上の可変状態）

を抱えたまま、最後に lambda を「追加」した。関数型に**なってしまった**のであり、
関数型として**設計された**のではない。

新 Ajisai は向きが逆である。正典ドラフト（`SPECIFICATION.html`）が既に規範化した
性質は、関数型言語の用語でそのまま言い直せる:

| 正典ドラフトの規範 | 関数型の用語 |
| --- | --- |
| セル評価は効果なし（§6.3）・決定論（§6.5） | 参照透過性。`RAND`/`NOW`/`INDIRECT` 型の語は最初から存在しない |
| 再計算は順序・並列度に非依存（§9.3） | 合流性（Church-Rosser 的性質）が規範要件 |
| セル＝引数なしの純粋な定義、ドキュメント＝相互参照する定義の系（§9.1） | プログラム＝相互再帰的な純粋束縛の集合。再計算＝その最小不動点の再構成 |
| NIL の passthrough（§4.5.1） | Maybe 的な失敗伝播（理由つき） |
| U と K3 論理（§7.5） | 三値論理の真理値がファーストクラス |
| 値は不変、書式・結合は値の外（§13.2） | イミュータブルな値意味論 |

つまり新 Ajisai は**公理として**純粋関数型であり、残る問いは一つだけである:
「ユーザーが**関数を抽象する手段**を何にするか」。Excel の答えが `LAMBDA` だった。

## 2. なぜ LAMBDA を導入しないか

1. **変数束縛の持ち込み。** `LAMBDA(x, y, ...)` は式言語に名前束縛・スコープ・
   シャドーイング・クロージャ環境を持ち込む。これらは表計算ユーザーが 40 年間
   一度も必要としなかった概念であり、「セル参照だけが名前である」という表計算の
   認知モデルを壊す。Excel がそれでも導入したのは、式が文字列であり
   セル参照以外の抽象手段を持たなかったからである。
2. **連接（後置）言語には元々 lambda が要らない。** Forth/Joy/Factor の系譜では、
   合成は並置（`F G` は F∘G）、無名関数は quotation（Ajisai の `{ ... }`）であり、
   引数は名前でなく位置（スタック）で受ける。ポイントフリーが既定である。
   新 Ajisai の canonical 形は後置なので、この資産をそのまま使える。
3. **コンセプトからの逸脱。** 「純粋関数型を前提に表計算を再定義する」のに、
   Excel が後付けした装置を後付けするのでは再定義にならない。

ただし quotation だけでは「**名前つきの多引数関数をどう定義し、どう見せるか**」に
答えていない。`{ 2 MUL } 'DOUBLE' DEF` はテキストであり、表計算の WYSIWYG からは
外れた場所（Script 面）に住む。Ajisai 固有の答えが必要で、それが次節である。

## 3. 提案: 関数＝表（関数表 / Function Table）

> **n 引数関数の定義とは、n 本の引数列と出力列を持つ表を作ることである。**

### 3.1 構成

- 表は任意の列を**引数列**として宣言できる（引数列はセル値を持たず、呼び出し時に
  束縛される）。引数列を 1 本以上持つ表が**関数表**である。
- 残りの列は通常の列式（正典ドラフト §5.5）で計算される。そのうち 1 本を
  **出力列**に指名する（既定は最右の式列）。
- 呼び出しは既存の修飾参照の自然な拡張:
  - canonical: `3 4 TRIANGLE@AREA` — 引数列に 3, 4 を束縛して AREA 列の式を評価
  - infix: `=TRIANGLE.AREA(3, 4)`、既定出力列なら `=TRIANGLE(3, 4)`
- 評価は通常のセル評価と同じ規則（純粋・決定論・予算つき）。関数表の列式が
  他のセル・表を参照していれば、呼び出し元はそれらに推移的に依存する
  （依存グラフ §9.1 にそのまま乗る）。

### 3.2 例示行 — 定義・テスト・メモ化の三位一体

関数表の本体（行）には**具体的な引数の行**を並べられる。各行は普通の行として
ライブに計算される。これが三つの役割を同時に果たす:

1. **WYSIWYG な関数定義。** 関数の中身が「例に対して動いている状態」で常に見える。
   中間列（補助計算）も隠れない。`LAMBDA` が失った可視性を取り戻す。
2. **組み込みテスト。** 期待値列（`EXPECT`）を置き、`EXPECT` と出力列の一致を
   テーブル不変量（`CONSERVE` と同じ機構、正典ドラフト §10.2）として宣言すれば、
   例示行はユニットテストになり、関数を壊す編集は再計算の瞬間に赤くなる。
3. **メモ化表。** 呼び出し引数が例示行に一致すれば、実装は行の値をそのまま
   返してよい（純粋性 §6.3 により意味論は同一）。対数表・三角関数表という
   「関数＝表」の 400 年の伝統は、そもそもメモ化の伝統である。

### 3.3 再帰と計算能力

関数表は自表を参照できる（`FACT` 表の出力列が `n-1` で `FACT` を呼ぶ）。
セル毎の実行ステップ予算（§6.4）の下で停止が保証されたまま、`LAMBDA` なしで
チューリング完全に到達する。相互再帰は表同士の相互参照であり、依存グラフ上の
**静的な**循環検出（§9.4）は「セル値の循環」にのみ適用し、関数表の呼び出し再帰は
動的予算で制御する（この区別は正典昇格時に §9.4 へ明文化する）。

### 3.4 高階と quotation の位置づけ

- 範囲語（`FOLD` `COUNT-IF` `SUM-IF` …）には **quotation か関数表参照のどちらでも**
  渡せる: `[SALES] { 0.1 MUL } FOLD` / `[SALES] TAX FOLD-WITH`。
- quotation `{ ... }` は「その場の小さな変換」用のインライン形、関数表は
  「名前を持ち、見えて、テストされ、再利用される」主要形。役割が重ならない。
- quotation は束縛を持たない（ポイントフリー）。名前が欲しくなったら、それは
  関数表に昇格すべき兆候である — という成長曲線を言語が示す。

### 3.5 先行研究と独自性（正直な位置づけ）

「表で関数を定義する」発想自体には学術的先行がある。Peter Sestoft の
**sheet-defined functions**（Funcalc, *Spreadsheet Implementation Technology*, 2014）は
シート上で関数を定義・コンパイルする処理系を実証しており、実装可能性の証拠になる。
Ajisai の独自性はその上に載る組み合わせにある:

1. 純粋性・決定論・順序非依存を**公理**とする正典の上での関数表（Funcalc は Excel 型
   意味論の上の拡張）。
2. **連接後置コア**との融合 — 引数列は「点自由な世界に付ける名前つきの入口」であり、
   式の内側には変数が最後まで存在しない。
3. **例示行＝テスト＝メモ化**の三位一体（§3.2）。
4. 厳密実数（連分数）と QUANTIZE/CONSERVE の上で動くこと。

Excel の `LAMBDA` とは出自も見た目も異なり、「表計算ソフトの再定義」という
コンセプトの内側から出てくる抽象手段である。

## 4. 構文決定: 丸括弧を連分数から解放する

旧正典および現行ドラフト §4.2.3 は、連分数の canonical serialization に
右ネストの丸括弧形 `( a0 ( a1 ( a2 ... )))` を使っていた。表計算ユーザーにとって
丸括弧は「infix のグルーピングと関数呼び出し」のための記号であり（`=SUM(A1:A10)`、
`=(A1+B1)*C1`）、この期待を最優先する。

**決定（正典ドラフトへ同時反映）:**

1. 丸括弧は infix 表面のグルーピングと関数呼び出し**専用**とする。
2. 連分数の canonical serialization は数学の標準記法（Khinchin 記法）へ変更する:

   ```
   [ a0 ; a1 , a2 , ... , a_{n-1} ]        （有限 = 有理数）
   [ 1 ; 2 , 2 , 2 , ... ]                  （遅延 = 表示予算で打ち切り、末尾 ... ）
   ```

3. この形は従来どおり**ソース構文ではない**（ドラフト §3.4 不変）。表示・診断・
   プロトコル出力にのみ現れ、打ち切り表示の再パースは禁止のまま。
4. 列参照 `[@PRICE]` / `[PRICE]` は数式内、CF 記法は出力面にのみ現れるため
   構文衝突は生じないが、区別は視覚的にも明瞭（CF は必ず先頭が整数で `;` を含む）。

## 5. 未決定事項（レビューで判断を仰ぐ）

1. 引数列宣言の表面（ヘッダ上の記法か、列プロパティか）。
2. 既定出力列の規則（最右の式列で良いか、明示必須か）。
3. `EXPECT` 期待値列（§3.2-2）を正典に含めるか、ホスト機能に置くか。
4. 初期リリースの高階語に quotation を含めるか、関数表参照のみで始めるか。
5. Script 面の DEF ワードの位置づけ — 関数表が主要な抽象手段になった後、
   DEF は連接合成の小道具として残すか、段階的に縮小するか。
