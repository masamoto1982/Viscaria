Viscaria Language Specification

Viscaria is the table-oriented Ajisai: an AI-first, exact-real spreadsheet language. This document is the single canonical design authority for the language. It descends from the vector-oriented Ajisai (the Ajisai repository), which is retained there for historical reference; the migration inventory is in Section 15. The language itself is still named Ajisai throughout this specification; Viscaria is the name of this project and its application.

Appendix A. Water on the Grid (non-normative index)

1. Language Identity

Ajisai is an AI-first, table-oriented, exact-real spreadsheet language.

A program is a document: a set of named finite tables laid out on canvases, together with a user word dictionary. Evaluation is recalculation: cells hold formulas, formulas reference other cells, ranges, and words, and a change propagates through the dependency graph deterministically (Section 9).

Flat Cell principle (normative). Structure has exactly one primary home: the grid. A cell holds an atomic value only (Section 4.1) — there are no nested sequence values. Collections are expressed by rows, columns, ranges, and tables; nested data is expressed as separate tables joined by key references.

Every numeric value is an exact real, represented internally as a (possibly lazy) continued fraction (Section 4.2). Finite continued fractions cover the rationals; lazy continued fractions cover the algebraic irrationals admitted by the runtime (e.g. SQRT). All arithmetic is performed on the continued-fraction representation directly, without intermediate rounding.

Runtime stack:

  • Rust interpreter core (value model, formula evaluation, table store, recalculation)
  • WASM boundary
  • TypeScript GUI/runtime shell (grid presentation, editing, persistence)

Ajisai is designed for mechanical reasoning, automated refactoring, and structurally searchable implementation. The dependency graph, cell diagnostics, and formula bodies are machine-readable at all times; an AI agent can read and repair a document without screen-scraping the grid.

The water metaphor of the previous canon — flow, vessels, bubbles, stagnation, water levels, gates, conservation — carries over intact and is indexed against the normative mechanisms of this document in Appendix A.

2. Specification Authority

2.1 Canonical

This document is the single canonical design authority for the Ajisai language as implemented by Viscaria: value model, formula semantics, table model, recalculation semantics, error model, and conformance requirements. Statements elsewhere (README, Reference, docs/dev/, code comments) are non-canonical.

2.2 Non-canonical

Display strings, GUI layout, presentation styling, host persistence formats, and any text marked "non-normative" are not part of language semantics. They may change without a specification change, subject to the constraints of Section 13.

2.3 Semantic Firewall

Observable semantics are defined exclusively in terms of values, their interpretation roles and capabilities (Section 12), and the machine-readable protocol fields named in this document. Internal representation choices (which scalar representation backs a value, how a table is stored, how the dependency graph is indexed) are never observable and must not be branched on by words or by external consumers. Human-readable display text is non-canonical and must not be used for machine decisions.

2.3.1.1 User-facing exactness transparency guarantee

For every program expressible with the Coreword set of this document, numeric observation is exact: comparisons of constructible values decide definitely (Section 7.4), display truncation is an observation limit rather than a new value (Section 4.2.3), and deliberate coarsening happens only through QUANTIZE with its residual preserved (Section 7.13). A user can rely on the grid never silently substituting an approximation for a value.

2.5 Authority order

  1. This specification.
  2. Machine-readable contract metadata generated from it (Section 7.14).
  3. Reference documentation and the GUI, which must not contradict 1–2.

2.6 Maturity

This specification is under active development: the implementation is being built up in phases (`docs/dev/ajisai-table-oriented-language-plan.md`), and sections describing words or surfaces not yet implemented are normative for the target, not a claim about the current build. Changes are proposed and reviewed like any specification change. Where a section is provisional it says so.

3. Formula Syntax

3.1 Token types

A formula is a sequence of tokens in canonical concatenative form, evaluated left to right (Section 6). Token types:

TokenExampleMeaning
Numeric literal42, 3/4, 1.25, -7An exact Scalar (Section 3.2)
Text literal'hello'A Text value (Section 3.3)
Cell referenceB3, TABLE2.B3The value of another cell (Section 3.5)
Column reference[@PRICE]Same-row value of a named column (Section 5.5)
Range referenceA1:A10, SALES.PRICE, TABLE2.A1:B5A rectangular region or a whole column, usable only where a Range is expected (Section 3.5)
WordADD, SUM, TAX-RATEBuilt-in or user word invocation
Code block{ 2 MUL }A formula-local executable block (Section 6.1); never a cell result
Definition / deletionDEF, DELUser word management (Section 8); valid only in the Script surface, not inside cells

3.2 Numeric literal formats

Numeric literals are convenience input forms for the same underlying exact representation (Section 4.2):

  • Integer: 42, -7
  • Fraction: 3/4 (reduced on read; denominator must be non-zero)
  • Decimal: 1.25 — read exactly as the rational 125/100 reduced; no floating-point intermediate is permitted

The literal style is input syntax only and is not retained for display (Section 4.2.3).

3.3 Text literals

Text literals are single-quoted: 'hello'. A quote inside text is escaped by doubling: 'it''s'. Text is an immutable sequence of Unicode scalar values; its equality is codepoint equality.

3.4 Serialized and display forms are not source syntax

The bracket continued-fraction serialization of Section 4.2.3 and every truncated display form are output surfaces only. They are not Ajisai source syntax, must not appear in formulas, and must not be parsed back as values.

3.5 Cell and range references

References name what they point at. Two sigils are orthogonal: . is the namespace path separator (document → table → member), and the bracketed [@…] marks a row-context reference ("this row's"). A cell's auto-name is its A1 coordinate, which is a position surface bound to the cell's stable identity (Section 5.3), so pointing at a cell the familiar A1 way requires no naming while the reference itself survives structural edits and merges.

  • Local cell: B3 — column letter(s) then row number, within the formula's own table; resolves to that cell's identity (Section 5.3).
  • Qualified cell: TABLE2.B3 — the namespace path TABLE.ADDRESS reaches any table in the document (Section 5.3).
  • Named cell (landmark): a cell given an explicit name (Section 5.3), referenced by that name — Total, or INVOICE.Total when table-qualified.
  • Column (same row): [@NAME] — the value in column NAME of the current row (the bracketed @ is the row-context marker); valid only in a column formula (Section 5.5).
  • Column / range: TABLE.NAME — a whole named column body, and B2:C5 / TABLE.B2:C5 — a rectangular region by two corner addresses. A Range is a reference, not a value (Section 4.1): it may appear only as the operand of a range-consuming word (Section 7.1) and can never be a cell's result.
  • Function-table application: TABLE.NAME, where NAME is a bare (unbracketed) output-column name of a function table, applies it to the preceding operands and yields that column's value (Sections 5.6, 6.6). A bare function-table name alone applies its default output column.

Disjoint by shape (normative). After a namespace path TABLE.MEMBER, the member is resolved by shape: a token matching the cell-address grammar (letters then digits, optionally a : range) is a cell or range; a bare name that is a data column is that column (a range); a bare name that is a function table's output column is an application when operands are present. A bare token matching the cell-address grammar inside a formula is a cell reference, never a word name; user words whose normalized name matches the address grammar cannot be defined (Section 8.2).

3.6 Infix surface form

The canonical form of a formula is the concatenative token sequence of Section 3.1. Implementations must additionally accept and display the infix surface form, a projection aimed at spreadsheet users:

Infix surfaceCanonical form
=A1+B1A1 B1 ADD
=SUM(A1:A10)A1:A10 SUM
=[@PRICE]*[@QTY][@PRICE] [@QTY] MUL
=IF(A1>0, 'yes', 'no')A1 0 GT 'yes' 'no' IF
=TAX(B2)B2 TAX
=TRIANGLE.AREA(3, 4)3 4 TRIANGLE.AREA

Normative rules for the surface form:

  • The infix surface is a bijective projection: every infix formula maps to exactly one canonical token sequence, and every canonical sequence expressible in the infix grammar maps back to a unique normal-form infix rendering. Semantics are defined on the canonical form only.
  • Operator spellings map to Corewords: + - * / → ADD SUB MUL DIV; < <= > >= = <> → LT LTE GT GTE EQ NEQ; & (text) → CONCAT. Precedence is the conventional arithmetic precedence; parentheses group.
  • Function-call syntax NAME(arg1, arg2, …) maps to arg1 arg2 … NAME for any built-in word, user word, or function table (default output column), so all of them are callable from the infix surface without declaration. The dot form TABLE.COL(args…) selects a specific output column and maps to args… TABLE.COL (Section 3.5).
  • In the host grid, a cell text beginning with = is an infix formula; a cell text that parses as a numeric literal is a Scalar literal; any other text is a Text literal. This entry convention is a host rule (Section 13), not part of formula syntax.
  • A canonical sequence that has no infix rendering (e.g. one using code blocks) is displayed in canonical form; the surface layer must not invent lossy renderings.

3.7 Syntax constraints

  • Tokens are whitespace-separated in canonical form.
  • Word names are case-insensitive on input and normalized to upper case (Section 8.2).
  • A formula that fails to parse is a formula error: the cell enters the error state of Section 11.1 and no value (not even NIL) is produced.

4. Value Model

4.1 Cell value types

The result of a cell is exactly one atomic value:

TypeDescription
ScalarAn exact real number, represented internally as a (possibly lazy) continued fraction (Section 4.2)
BooleanA definite logical truth value, true or false (Section 4.3). Not a number.
TextAn immutable Unicode string (Section 4.4)
NILThe reason-carrying absence of a value (Section 4.5)
UnknownThe logical-undecidability truth value U (Sections 4.3, 7.4.1), storable as a cell result
EmptyThe deliberate blankness of an untouched cell (Section 4.6)

There are no composite cell values. Vector, Record, CodeBlock, and handle types of the previous canon are not cell values (Section 15). During formula evaluation two additional transient kinds exist — CodeBlock (Section 6.1) and Range reference (Section 3.5) — but a formula whose final result is transient is a formula error (Section 6.2).

4.2 Scalar: exact-real continued-fraction arithmetic

Carried over from the vector-oriented canon §4.2 unchanged in substance; only cross-references and word names are adapted (module-qualified MATH@SQRT/MATH@POW become the Corewords SQRT/POW, Section 7.3).

All numeric values are exact reals represented as continued fractions (CF). A scalar is a sequence of integer partial quotients \([a_0;\, a_1, a_2, \dots]\), finite for rationals and lazy (potentially infinite) for irrationals:

\[ [a_0;\, a_1, a_2, \dots] \;=\; a_0 + \cfrac{1}{a_1 + \cfrac{1}{a_2 + \cfrac{1}{\ddots}}} \]

4.2.1 Canonical form

A continued fraction is in canonical form when:

  1. \(a_0\) is any integer.
  2. \(a_1, a_2, \dots\) are strictly positive integers.
  3. If the sequence is finite and has length \(n \ge 2\), then the last partial quotient \(a_{n-1}\) is greater than 1 (no trailing 1).
  4. The sequence terminates iff the value is rational.

These rules give every real value a unique canonical CF, so equality of canonical CFs decides equality of values whenever the comparison terminates (Section 7.4).

4.2.2 Internal representation

A scalar is internally one of the following representations. Which representation is used is not observable; only the canonical CF sequence and its value are.

  • Rational — a finite CF, stored equivalently as either a small (i64, i64) reduced fraction or a (BigInt, BigInt) reduced fraction. Partial quotients are generated on demand by floor-division Euclidean algorithm.
  • AlgebraicSqrt — SQRT of a non-negative rational. The CF expansion is eventually periodic (Lagrange's theorem) and produced lazily.
  • Gosper — an unevaluated bihomographic transform \(\dfrac{a\,xy + b\,x + c\,y + d}{e\,xy + f\,x + g\,y + h}\) of one or two operand CFs, used by arithmetic (Section 7.3). Partial quotients of the result are emitted as soon as the next quotient is unambiguously determined by the current Möbius coefficients.
  • LazyCf — any other lazy CF stream (reserved for future words; not produced by the Coreword set defined in this document).

Möbius coefficients used by Gosper transforms are stored as arbitrary-precision integers (BigInt) at all times. Implementations must not use bounded-width coefficient storage that can overflow during normal evaluation.

4.2.3 Display and serialization

The canonical AI-readable serialization of a scalar is the standard mathematical (Khinchin) bracket form:

[ a0 ; a1 , a2 , ... , a_{n-1} ]

with the integer part before the semicolon and the remaining partial quotients comma-separated. A lazy infinite CF is serialized by emitting partial quotients up to an implementation-defined display budget and terminating with the marker ... before the closing bracket (e.g. [ 1 ; 2 , 2 , 2 , ... ]); the truncated display is non-canonical and must not be parsed back as an exact value.

Parentheses are reserved for the infix surface (grouping and function-call syntax, Section 3.6) and never appear in the CF serialization. The bracket form is not Ajisai source syntax (Section 3.4) and cannot collide with the column-reference forms of Section 3.5, which occur only in formulas while the CF form occurs only in output surfaces. It appears only in display and serialization output under the ContinuedFraction interpretation role (Section 12.2) and in AI-targeted diagnostics. Under the RawNumber role a rational scalar renders as a reduced numerator/denominator (Section 12.2); the surface literal style of Section 3.2 is convenience input syntax only and is not retained for display.

4.2.4 Equivalence of representations

Two scalars are equal as values iff they produce the same canonical CF sequence. A Rational scalar and a Gosper or AlgebraicSqrt scalar may compare equal whenever their generated partial quotients agree at every position. The internal representation tag is not part of value identity and must not be branched on by Corewords or external consumers.

4.2.5 Nearest-integer continued fractions (comparison expansion)

The comparison procedure of Section 7.4.1 expands operands as nearest-integer continued fractions. This is an internal expansion: the regular continued fraction of Sections 4.2.1–4.2.4 remains the sole basis for value identity, canonical form, display, and serialization. The nearest-integer expansion changes only how comparison consumes a value and, consequently, the unit of the comparison budget (Section 7.4.1.1).

The regular continued fraction (RCF) of Section 4.2.1 takes each partial quotient by floor, leaving a remainder in \([0, 1)\). The nearest-integer continued fraction (NICF) instead takes each partial quotient by rounding to the nearest integer, leaving a remainder in \((-\tfrac{1}{2}, \tfrac{1}{2}]\). An NICF is a semiregular continued fraction where each \(b_i\) is the nearest integer to the current tail, each \(\varepsilon_i \in \{+1, -1\}\) is the sign of the corresponding remainder, and \(b_i \ge 2\) for \(i \ge 1\). The NICF converges at least as fast as the RCF and on average meaningfully faster.

Tie-break (normative). Each partial quotient is \(b_i = \operatorname{round}(t)\) of the current tail \(t\), where round sends a value whose fractional part is exactly \(\tfrac{1}{2}\) down to the lower integer; equivalently the post-step remainder lies in the half-open interval \((-\tfrac{1}{2}, \tfrac{1}{2}]\). This makes the NICF digit sequence of every value deterministic, so two conforming implementations agree on it.

NICF changes which expansion the comparison procedure consumes (Section 7.4.1.1). It does not change value identity or canonical form (which remain the RCF, Sections 4.2.1, 4.2.4), nor display and serialization (Section 4.2.3; NICF digits are never serialized), nor the non-observability of the internal representation tag. What NICF does make observable, indirectly, is the unit of the comparison budget: a budget term, and hence an agreedPrefix count (Section 4.5.0), is a semiregular (NICF) term.

4.2.6 Numeric error policy

Ajisai's exactness requirement is not limited to avoiding floating-point rounding error. A conforming implementation must not silently turn numeric work into hidden value corruption through information loss, loss of significance, cancellation, truncation error, or overflow.

  • Rounding and truncation. Core numeric operations preserve exact values. Any finite display or observation budget is an observation limit only; it is not a new approximate value and must not be parsed back as canonical data.
  • Deliberate quantization. The only operation that maps an exact value onto a coarser grid, QUANTIZE (Section 7.13), is explicit in its step and emits the exact residual alongside the quantized value; the two reconstruct the input exactly (\(q + r = x\)), so quantization is a visible, lossless-in-aggregate operation rather than a silent approximation.
  • Information loss and cancellation. Algebraically equivalent values remain represented by exact scalar structures rather than by rounded samples, so subtracting near-equal values or combining large and small terms must not discard significant information as an unreported side effect.
  • Overflow. Internal integer state that participates in scalar identity or arithmetic, including Gosper/Möbius coefficients, uses arbitrary precision or an equally exact strategy. If an implementation-specific resource limit is reached, the result is a diagnosed execution/resource failure, not wraparound.
  • Undecidable observation. When a comparison or other observation cannot decide within its budget, the result is the logical UNKNOWN (Section 7.4.1), not a rounded Boolean guess.

4.2.7 Admitted exact-real domain of the Coreword set

Definition (admitted domain \(D\)). \(D\) is the smallest set of real numbers that

  • contains every rational (\(\mathbb{Q} \subseteq D\));
  • contains \(\sqrt{r}\) for every non-negative rational \(r\) — the values SQRT produces (Sections 4.2.2, 7.3); and
  • is closed under the four field operations \(+, -, \times, \div\) (ADD SUB MUL DIV, with division by a non-zero element).

Equivalently, \(D\) is the multiquadratic closure of \(\mathbb{Q}\). Every element of \(D\) has a unique normal form as a finite \(\mathbb{Q}\)-linear combination \(\sum_d c_d \sqrt{d}\) over distinct square-free integers \(d \ge 1\). This normal form is a representation detail (Sections 2.3, 4.2.4) and is not observable.

Closure and boundary of the Coreword set.

  • ADD SUB MUL DIV keep \(D\) closed. Division by exact zero bubbles to NIL per Section 11.2; no element of \(D\) other than \(0\) is indistinguishable from \(0\).
  • MOD FLOOR CEIL ROUND and the six comparison relations are total on \(D\): the sign, integer part, and order of any \(D\)-element are exactly computable in finitely many steps.
  • QUANTIZE (Section 7.13) keeps \(D\) closed and is total on \(D\) given a positive rational step.
  • SQRT is defined on the rational sub-domain only. SQRT of a non-rational \(D\)-element is malformed use → formula error (Section 11.2); SQRT of a negative rational is a well-formed domain miss → Bubble/NIL. \(D\) is not closed under SQRT, and the Coreword set constructs no nested radicals or transcendentals.
  • POW is defined for an integer exponent only, which keeps the result in \(D\); a non-integer exponent is malformed use.

Because \(D\) is exactly this multiquadratic field, equality and order over \(D\) are decidable exactly and totally (Section 7.4). The lazy continued-fraction machinery and the comparison budget exist for exact reals outside \(D\) (the values reserved by LazyCf) and for the user-facing COMPARE-WITHIN (Section 7.4.2); they never make a comparison of \(D\)-elements return unknown. This is the mechanism behind the transparency guarantee of Section 2.3.1.1.

4.3 Boolean and the truth domain

A Boolean is a definite logical truth value, true or false. It is a distinct value kind, not a number: TRUE is not the Scalar 1 and FALSE is not the Scalar 0, so TRUE 1 EQ is false. A Boolean carries the TruthValue interpretation role (Section 12.2) and renders uniformly as TRUE or FALSE (display-only, non-canonical).

The third truth value Unknown (U) is not a Boolean; it is the logical-undecidability value of Section 7.4.1, observed as truthValue = unknown. In the table-oriented value model U is storable as a cell result: a formula whose result is U displays the cell as undecided, with diagnosis.agreedPrefix available as diagnostic context (Section 4.5.0). Downstream cells consuming a U cell receive U and the K3 logic of Section 7.5 applies.

4.4 Text

An immutable sequence of Unicode scalar values, carrying the Text interpretation role. Text equality is codepoint equality; ordering is lexicographic by codepoint. Text participates in CONCAT and the text words of Section 7.6 and is never implicitly converted to or from Scalar; conversion is explicit (TO-NUMBER, TO-TEXT).

4.5 NIL

Carried over from the vector-oriented canon §4.5 unchanged in substance.

NIL represents the absence of a value. It is produced by well-formed operations that yield no meaningful result (Section 11.2).

4.5.0 Diagnostic absence metadata

NIL is a diagnostic absence value. NIL identity is separate from its reason, origin, recoverability, and diagnostic context. Surface display, equality, hashing, and serialization treat all NIL values uniformly unless a protocol explicitly asks for diagnostic metadata.

For any NIL value: semanticKind is absence; capabilities includes diagnosable and nilPassthrough; the human-readable display text NIL is non-canonical and must not be used for machine decisions.

NIL metadata is exposed as an optional structured absence object with these fields:

FieldMeaningMachine-readable contract
reasonDirect reason the value became NILOptional lower camel case protocol string
originPath by which the NIL was producedRequired lower camel case protocol string
recoverabilityUI/AI hint for next actionRequired lower camel case protocol string
diagnosisThree-layer debug diagnosis (when / where / why, plus summary, evidence, nextChecks, and the optional agreedPrefix of Section 7.4.1)Optional structured object

NIL? checks only whether a value is absent. It must not branch on absence.reason. Reason-specific code must use the explicit diagnostic accessors NIL-REASON, NIL-ORIGIN, NIL-RECOVERABLE?, NIL-DIAGNOSIS.

4.5.1 NIL passthrough

Operations classified as NIL-passthrough in Section 7 do not raise an error when a NIL operand is encountered. Instead, they produce NIL. If any operand consumed by the operation is NIL, the operation consumes its operands as it normally would and produces a single NIL result. The resulting NIL inherits the reason of the leftmost NIL operand that carried a reason, which makes the cause traceable through long dependency chains: a division by zero three tables away surfaces in the final cell with its original reason.

NIL-passthrough applies to arithmetic, comparison, the unary numeric rounding words, and range aggregation (Sections 7.1, 7.12). It does not apply to type-conversion words or to VENT (^) itself, whose entire purpose is to react to NIL.

4.5.2 NIL versus Unknown

NIL and the logical truth value Unknown (U, Section 7.5) are distinct and must not be conflated. NIL is an operational absence: a diagnostic bubble that records why a value is missing. U is a logical undecidability: a definite member of the three-valued truth domain that records that a proposition could not be settled true or false. When NIL and U meet in the same operation, NIL takes priority: the reason-bearing operational information is never erased by the logical value, unless an absorbing definite operand (false for AND, true for OR) settles the result first.

4.6 Empty

Empty is the value of a cell that has no content: never edited, or cleared. Empty is distinct from NIL:

  • Empty is deliberate blankness. It carries no diagnosis and no reason. It is the spreadsheet-conventional state of an untouched grid.
  • NIL is a failed computation. It carries diagnostic metadata (Section 4.5.0).

Normative behavior:

  • Range aggregations skip Empty cells (Section 7.1): SUM over a column with blanks sums the present values. COUNT counts non-Empty cells.
  • Direct reference to an Empty cell inside arithmetic, comparison, or logic is a well-formed domain miss and produces NIL with reason = emptyCell (Section 11.2) — silent coercion of blank to 0 is prohibited, because it is precisely the class of spreadsheet bug this language exists to remove. A formula that wants blank-as-zero writes it explicitly: A1 ^ 0 (Section 7.7).
  • EMPTY? tests Emptiness. NIL? of Empty is false.
  • A formula cannot produce Empty; only the host can set a cell to Empty by clearing it. Empty is a cell state, not a computable result.

5. Document and Table Model

5.1 Document

A document is the unit of persistence, naming, and sharing. It consists of:

  • a set of tables (Section 5.2), each assigned to one sheet;
  • a set of sheets — presentation canvases carrying table placement, ordered as tabs; and
  • one user word dictionary (Section 8), owned by the document.

Sheets are a presentation grouping only (Section 13.2): moving a table between sheets changes no reference and no value. The language core observes the document as "a set of tables plus a dictionary".

5.2 Tables

  • A table is a finite grid of cells with a fixed current size (rows × columns), growable by explicit host action.
  • Auto-named, explicitly renamable. Every table is auto-named on creation (Table1, Table2, …) and may be given an explicit name; either way the name is unique within the document and normalized like word names (Section 8.2). A table name is a namespace (Section 3.5). A rename changes the name attached to the table's identity; references bind to identity (Section 5.3), so no formula text is rewritten.
  • Columns are likewise auto-named (from the header row's Text cells when present, otherwise ColA, ColB, …) and explicitly renamable; column names are unique within their table and enable the @NAME / TABLE.NAME reference forms (Section 3.5).
  • Each column may carry a column formula (Section 5.5) and a format (Section 10.1); each table may carry invariants (Section 10.2).

5.3 Identity, auto-names, and references

Universal auto-naming with explicit override. Every entity — document, sheet, table, column, and cell — is auto-named on creation and may be given an explicit name. Each also has a stable identity assigned at creation and never reassigned by insertion, deletion, merge, or move. The identity is the canonical referent; the name is how a formula spells it. This keeps the A1 convention's chief benefit — you never have to invent a name to point at a value — while removing its fragility.

  • Cell auto-name = A1 coordinate, bound to identity. A cell's auto-name is its A1 coordinate (column letters A–Z, AA–…, then 1-based row). But a reference written as a coordinate binds to the cell's identity, not to the grid slot, and is rendered at the cell's current coordinate. So inserting or deleting rows/columns re-renders existing references at their new coordinates without rewriting them, and a reference never silently shifts to a different value.
  • Deletion. A reference to a cell that is actually deleted becomes a formula error (Section 11.1) — it does not resolve to whatever later occupies the slot.
  • Merges. A merged region is one cell — one identity — occupying a rectangle (presentation span, Section 13.2). Covered coordinates have no identity of their own; a coordinate inside a merge resolves to the merged cell. There is no "hole" to patch.
  • Explicit names. An explicit name overrides the auto-name as the preferred spelling and, for a cell, makes it a landmark referable as Total (or TABLE.Total); for a table or sheet, it is the namespace label.
  • Qualification. The namespace path TABLE.ADDRESS (and TABLE.NAME, TABLE.B2:C5) reaches any table in the document; within a table's own formulas the qualifier is optional (Section 3.5).

The A1 coordinate is thus a presentation surface over a stable identity, exactly as the infix form is a surface over the canonical formula (Section 3.6) and display rounding is a surface over the exact value (Section 10.1). The identity, not the coordinate, is what persists and what the machine-readable document format records (Section 13.3).

5.4 Ranges are references, not values

A Range denotes a rectangular region of one table. It exists only during formula evaluation as the operand of a range-consuming word (Section 7.1). It cannot be stored in a cell, returned by a formula, or bound as a user-word result. This closes the structural ambiguity that sank the previous design: there is no value-level sequence for a grid region to be confused with.

Iteration order over a range, where a word needs one (e.g. FOLD), is row-major (left to right, then top to bottom) and normative.

5.5 Column formulas

A column may carry a single formula that defines the value of every body cell in that column. It is the table-oriented successor of the previous canon's MAP:

  • The formula is evaluated once per row, in a row context that resolves [@NAME] to the same-row cell of column NAME.
  • Row evaluations are mutually independent: a column formula must not reference other rows of its own column, and may reference other columns of the same row, whole ranges of other columns or tables, and words. Independence makes per-row evaluation order unobservable and row-parallel evaluation semantics-preserving (Section 9.3).
  • A cell in a column governed by a column formula cannot also hold an individual formula; the host may allow a per-cell override that detaches the cell from the column formula, and the detachment is machine-visible.

5.6 Function tables

A table may declare one or more columns as parameter columns. A table with at least one parameter column is a function table: it defines a pure function whose arguments are the parameter columns in left-to-right column order. Defining an n-ary function in Ajisai means building a table with n parameter columns — there is no lambda form and no variable binding in the formula language (docs/dev/ajisai-functional-foundations.md, non-normative rationale).

  • Declaration. Parameter status is machine-readable column metadata. The host presents it on the header (presentation choice, Section 13.2); the metadata, not the presentation, is normative.
  • Body. Every non-parameter column of a function table is governed by a column formula (Section 5.5). Its formulas may use [@PARAM] for the parameter columns and [@COL] for the other formula columns of the same table, and may reference cells, ranges, and tables elsewhere in the document.
  • Output column. One formula column is the default output column: the rightmost formula column unless the table designates another (machine-readable metadata). A call may also name any formula column explicitly (Section 6.6) — a function table with several formula columns is thereby a function with several named results, one application per named column.
  • Example rows. The body rows of a function table are example rows: each row supplies literal values in the parameter columns (and in the EXPECT column, Section 10.3, if present), and its formula columns evaluate as ordinary rows under Section 5.5. Example rows are simultaneously the visible, live definition of the function (intermediate columns included), its test fixture (Section 10.3), and a memo table (Section 6.6). A function table may have zero example rows.
  • Namespace. Table names and user word names share one namespace: a table may not be named like an existing word and vice versa (Section 8.2). Both are document-unique.

6. Formula Evaluation

6.1 Evaluation context

A formula evaluates its canonical token sequence left to right over a fresh, empty operand stack, with access to: the built-in Corewords (Section 7), the document's user dictionary (Section 8), the values of referenced cells and ranges (resolved through the recalculation engine, Section 9), and — inside a column formula — the row context (Section 5.5). The operand stack is formula-local and invisible outside evaluation; it is an evaluation mechanism, not a language surface.

Code blocks { … } are transient values usable by higher-order Corewords within the formula (e.g. FOLD, Section 7.1). They cannot escape the formula.

6.2 Single-result rule

A formula's evaluation must end with exactly one value on the operand stack, and that value must be atomic (Section 4.1). Zero results, multiple results, or a transient result (CodeBlock, Range) is a formula error (Section 11.1).

6.3 Purity

Cell evaluation is effect-free. No word invocable from a formula performs input, output, host communication, or mutation of the document (a formula cannot write another cell, define a word, or alter a table). Effects exist only as host commands outside cell evaluation (Section 13.1). This makes recalculation free to evaluate cells in any dependency-respecting order, in parallel, or speculatively, with no observable difference (Section 9.3).

6.4 Budgets

Each cell evaluation runs under an implementation-defined execution step budget, and each comparison under the comparison budget of Section 7.4.1. A cell that exhausts its step budget produces NIL with reason = stepBudgetExhausted and a diagnosis; one runaway cell must not block the recalculation of unrelated cells.

6.5 Determinism

Formula evaluation is deterministic: the same document state produces the same cell values, diagnostics, and truth values on every conforming implementation. There is no observable randomness, wall-clock, or evaluation-order dependence.

6.6 Function-table application

Naming a function table applies it, exactly as naming a word invokes it (concatenative rule: naming is execution; deferral is quotation). For a function table F with parameter columns \(p_1, \dots, p_n\) (left-to-right) and a formula column COL, the application a1 … an F.COL:

  1. consumes the top n operands, binding them positionally to \(p_1, \dots, p_n\); fewer than n available operands is malformed use → formula error;
  2. evaluates COL's column formula in a call context in which [@PARAM] resolves to the bound argument and [@X] for a sibling formula column X evaluates X's formula in the same call context (on demand; each sibling at most once per application);
  3. produces the single resulting value. The result must be atomic (Section 4.1); the single-result rule of Section 6.2 continues to apply to the calling formula as a whole.

The bare form a1 … an F applies the default output column (Section 5.6).

  • Purity and budgets. A call is ordinary evaluation: pure (Section 6.3), deterministic (Section 6.5), and metered against the calling cell's step budget (Section 6.4).
  • Arguments are values. Arguments may be any atomic value, including NIL and U; no implicit passthrough applies at the call boundary — the body's words handle absence per their own contracts. Transient kinds (Range, CodeBlock) are not passable arguments; supplying one is malformed use → formula error.
  • Dependencies. The calling formula depends on the function table's definition (its column formulas, parameter declarations, and output designation) and, transitively, on every cell, range, and word the evaluated body references (Section 9.1). Example-row values are not dependencies of a call, except through the memo rule below.
  • Memoization by example rows (normative soundness condition). An implementation may answer an application from an example row whose parameter values equal the arguments, provided the equality of every argument with the row's parameter value is decided exactly (definite per Section 7.4 — always the case within the admitted domain \(D\), Section 4.2.7). If any such equality would be undecidable, the memo must not be used and the body is evaluated. By purity, the memoized answer and the evaluated answer are the same value; memoization is never observable except through resource use.
  • Higher-order use. To pass a function rather than apply it, wrap the application in a quotation: SALES.AMOUNT 0 { TAX ADD } FOLD. There is no function-reference value kind.
  • Recursion. A body may apply its own table or tables that apply it back. Call recursion is dynamic and bounded by the step budget; it is not a reference cycle (Section 9.6).

7.4 Comparison

Carried over from the vector-oriented canon §7.4 with the stack-mode (STAK) and modifier material removed; the decidability core is unchanged.

CanonicalInfix sugarDescription

Comparisons return a truth value with the TruthValue interpretation role: true or false, or — when the comparison is not decidable within the comparison budget — unknown (U, the third value of the three-valued logic of Section 7.5).

Exactness over the admitted domain (normative). Over the admitted domain \(D\) of Section 4.2.7 — every value the Coreword set can construct — the six relations are total and exact: each returns a definite true or false and never unknown. The U outcome is reserved for exact reals outside \(D\) and for the explicit-budget word COMPARE-WITHIN (Section 7.4.2). For example =SQRT(2)=SQRT(2) evaluates to TRUE with no budget in play (Section 2.3.1.1).

The set of comparison primitives is intentionally complete (all six standard ordering relations), so that an automated producer can emit the relation that matches its intent directly. Every relation is independently registered with contract metadata (Section 7.14) and is NIL-passthrough (Section 7.12). Comparison across value kinds (Scalar vs Text, etc.) is a well-formed domain miss → NIL, except EQ / NEQ, which are total across kinds (values of different kinds are simply unequal).

7.4.1 Decidability and comparison budget

Comparison on continued fractions proceeds by emitting partial quotients of the two operands in parallel and stopping at the first index where they differ. For two finite CFs this always terminates. For lazy CFs, two distinct irrationals always differ at some finite index, but two equal irrationals never differ — the procedure does not terminate by itself.

To preserve totality, every comparison runs under an implementation-defined partial-quotient budget. If the budget is exhausted before a difference is found:

  • the comparison produces the truth value unknown (U), observed as truthValue = unknown;
  • U is a logical truth value, not an operational absence: the comparison does not produce a reason = undecidable NIL, and the Bubble Rule (Section 11.2) does not apply to it;
  • U flows into the three-valued logic of Section 7.5 directly, settled by the Kleene truth tables.

The agreed-prefix length — the number of leading terms that matched before the budget was exhausted — is carried on the U result in the machine-readable diagnosis.agreedPrefix field (Section 4.5.0). It is diagnostic context only and does not change the observable truthValue. The budget value itself is not part of observable semantics; it must be high enough that distinct rationals always decide.

7.4.1.1 NICF-accelerated comparison

The comparison procedure decides the order of two operands by emitting their nearest-integer continued fractions (NICF, Section 4.2.5) in parallel. The order computed over NICF is the true order of the values: NICF never flips a decided order; it only moves the budget boundary at which a pair of distinct values crosses from U to decided, in the favorable direction.

Budget unit and agreedPrefix. One budget term is one semiregular (NICF) term, uniformly for the six relations and for COMPARE-WITHIN. The agreedPrefix of a U result counts matching NICF terms; it is a non-negative integer, monotone in the budget, 0 for operands that differ at the first term, and diagnostic-only. Because the budget unit is user-observable through COMPARE-WITHIN, NICF expansion is a conformance requirement for every comparable scalar representation (Section 4.2.2), not an optional optimization.

7.4.2 Explicit-budget comparison: COMPARE-WITHIN

Signature: a b budget → -1 | 0 | 1 | U.

COMPARE-WITHIN consumes two numeric values and a positive integer budget, and emits partial quotients of \(a\) and \(b\) in parallel for at most budget steps. It produces the scalar -1, 0, or 1 (role RawNumber) as the exact sign of \(a - b\) when the order settles, and the logical U — carrying diagnosis.agreedPrefix equal to the consumed budget — when it does not. For two rational operands the comparison always decides regardless of budget. It is NIL-passthrough for \(a\) and \(b\); a non-positive or non-integer budget, or a non-numeric operand, is malformed use → formula error. The implicit budget of the bare relations is an implementation-defined constant; COMPARE-WITHIN is the only way to observe or override the depth.

7.4.3 Propagation of U through comparison-dependent words

The common rule is U-honesty: when the comparison a word relies on is undecidable, the word must surface that undecidability rather than fabricate a decision. No word may treat U as true, as false, or as a malformed-input error.

  • MIN and MAX (Section 7.1) select by order. When a governing comparison does not decide within the budget, the result is U (carrying agreedPrefix) — the program cannot be told which cell is extremal when the order is unknown. They remain NIL-passthrough; NIL takes priority over U per Section 4.5.2. They short-circuit on the first undecidable pair in range order.
  • IF and COND (Section 7.7): a guard that yields U is not a definite true, so its branch does not fire; see Section 7.7 for the outcome rules.
  • COUNT-IF / SUM-IF (Section 7.1): a U predicate makes the whole aggregate U.

7.13 Explicit quantization and residual

Carried over from the vector-oriented canon §7.13 unchanged in substance; the stack-effect header is restated as a signature.

The core numeric words keep values exact (Section 4.2.6); the only operation that deliberately maps a value onto a coarser grid is QUANTIZE. It makes that projection explicit in both its step and its outcome, and — unlike FLOOR, CEIL, and ROUND — it does not discard the fraction it rounds away. The discarded amount is emitted as an exact residual, so quantization loses nothing that is not deliberately dropped afterward.

Signature. x step → q r, where x is the scalar to quantize; step is a strictly positive rational quantum (for example 1/100 for a two-decimal cents grid); q is the quantized value, an exact integer multiple \(q = n \cdot \mathit{step}\); and r is the exact residual \(r = x - q\). (In a cell formula, the two results are consumed by a following word — commonly the format machinery of Section 10.1 or an explicit ADD/drop; a bare QUANTIZE as the whole formula violates the single-result rule of Section 6.2.)

Residual identity. For every well-formed call \(q + r = x\) exactly. Because x, q, and r are all exact, the pair reconstructs x with no loss; silent loss of significance cannot occur.

Rounding rule. q is the multiple of step nearest to x; a tie is resolved to the even multiple (banker's rounding). Unlike ROUND, whose ties break away from zero, it introduces no systematic bias when many half-way values are quantized, which is why it is the appropriate default for currency.

NIL, error, and decidability. A NIL x passes through to both outputs, carrying its reason (Section 7.12). A step that is not a strictly positive rational is malformed use → formula error, mirroring the deliberate asymmetry of MOD (Section 7.3). For x in the admitted domain \(D\) (Section 4.2.7) and rational step, both outputs are exact rationals computable in finitely many steps, so QUANTIZE is total on \(D\); for an x outside \(D\) whose nearest multiple cannot be decided within the comparison budget, both outputs project to an Undecidable Bubble/NIL (Sections 7.4.1, 11.2) rather than guessing.

Worked example. Quantizing the exact share 100/3 to cents: =QUANTIZE-VALUE(100/3, 1/100) yields 3333/100 with residual 1/300 retained in the cell diagnostics (Section 10.1); \(q + r = 100/3\) exactly, so an allocation of a total across several parts reconciles to the total with no invented or lost fraction of a cent.

7.14 Coreword contract metadata

Every Coreword carries machine-readable contract metadata: arity and result count (mass), NIL policy (nil_policy: passthrough / rejects / projecting), U policy (U-honest words, Section 7.4.3), and domain notes. The metadata is generated from this specification and is the second authority tier (Section 2.5). Automated tooling uses it to validate formulas, drive completion, and check test coverage (Section 16).

8. User Words

8.1 Definition

A user word is defined in the Script surface (Section 13.1), not inside a cell:

{ 8 100 / MUL } 'TAX' DEF

defines TAX; a cell may then use =TAX(B2) (infix) or B2 TAX (canonical). 'NAME' DEL removes a definition.

8.2 Rules

  • Word names are case-insensitive, normalized upper case. A name matching the cell-address grammar (Section 3.5), colliding with a Coreword, or colliding with a table name (Section 5.6) cannot be defined; table renames are refused symmetrically when they would collide with a word.
  • The dictionary belongs to the document (Section 5.1). Words see the same evaluation context as formulas (Section 6.1) and are subject to the same purity rule (Section 6.3).
  • A word body may leave any number of values for its caller (words compose on the operand stack); the single-result rule (Section 6.2) applies to the formula, not to each word.
  • Words may reference cells and ranges. Such references participate in the dependency graph (Section 9.1): a cell using the word depends, transitively, on the cells the word reads.
  • Recursion is permitted, bounded by the per-cell step budget (Section 6.4).

8.3 Redefinition

Redefining a word invalidates every cell that depends on it (directly or through other words); those cells recalculate under Section 9. Redefinition is atomic: no recalculation observes a half-updated dictionary.

8.6 Word identity and content addressing

A word's identity for caching and diffing purposes is the content address of its normalized body. Two words with identical bodies share a content identity; the identity is machine-readable but not observable from formulas.

9. Recalculation

9.1 Dependency graph

The runtime maintains, for every cell and word, the set of cells, ranges, and words its formula references (forward edges) and the inverted index of dependents (reverse edges). A range reference creates edges from every cell in the range. The graph is machine-readable at all times (Section 14).

9.2 Dirty closure and minimal recalculation

When a cell's formula or literal changes, a word is (re)defined, or a structural edit occurs, the runtime computes the dirty set: the transitive closure of dependents of the changed nodes over reverse edges. Only dirty cells are re-evaluated. Cells outside the dirty set must not be re-evaluated in a way that is observable (caching is an implementation choice; the values must be identical either way).

9.3 Evaluation order and determinism (normative)

Dirty cells are evaluated in a topological order of the dependency graph restricted to the dirty set. Cells with no dependency relation between them may be evaluated in any order, serially or in parallel: because cell evaluation is pure (Section 6.3) and deterministic (Section 6.5), the resulting document state is independent of the chosen order and degree of parallelism. This independence is normative; an implementation whose results can differ by scheduling is non-conforming.

9.4 Reference cycles

A dependency cycle among cells (directly or through words and ranges) is detected during recalculation planning. Every cell on the cycle produces NIL with reason = referenceCycle and a diagnosis identifying the cycle's members. Cells that depend on the cycle receive the NIL by passthrough (Section 4.5.1). Cycle detection is exact: cells not on any cycle are unaffected. There is no iterative- calculation mode in this draft.

9.5 Budget isolation

Step and comparison budgets (Section 6.4) are per-cell. A budget exhaustion in one cell yields that cell's NIL/U outcome and normal propagation; it must not abort the recalculation of cells that do not depend on it.

9.6 Call recursion versus reference cycles

Section 9.4's static cycle detection governs cell-value dependencies: a cell (or example-row cell) whose value transitively requires its own value. It does not apply to function-table call recursion (Section 6.6): a function table whose body applies itself (or a mutual group of tables applying each other) is well-formed, because each application binds fresh arguments and is bounded dynamically by the calling cell's step budget — exhaustion yields NIL with reason = stepBudgetExhausted, not referenceCycle. An example row whose own evaluation applies its table with arguments equal to that same row's parameters (a self-memo loop) is a genuine value cycle and is diagnosed under Section 9.4.

10. Value Integrity

10.1 Cell and column format is quantization

A numeric format on a cell or column (e.g. "two decimals", "satoshi") is not a display-only skin. Normatively, a format with grid step s means the displayed value is QUANTIZE of the exact value with step s (Section 7.13), and:

  • the exact value remains the cell's value — references, aggregation, and export read the exact value, never the displayed one;
  • the quantization residual is preserved in the cell's machine-readable diagnostics, and the host must make "display differs from exact value" perceivable (Section 13.2);
  • a format never changes semantics: removing a format changes only presentation.

The word QUANTIZE-VALUE (x step → q, the residual moved to diagnostics) is provided for formulas that want the quantized value as their result while keeping the residual observable; it is definitionally QUANTIZE plus residual-to-diagnostics and shares its totality and error rules.

10.2 CONSERVE: table invariants

Carried over from the vector-oriented canon §13.3, retargeted from vector parts to range parts.

Value conservation is the property that a transform which claims to preserve a numeric total preserves it exactly — the same law double-entry bookkeeping enforces continuously, lifted from ledgers to the grid. It is not statically inferable in general, so it is exposed as an explicit guard.

Word form. total range CONSERVE → true: when the exact sum of the range's numeric cells (Empty skipped, Section 7.1) equals total, the result is true. On a mismatch, on any NIL cell in the range, on a NIL total, or on an equality that cannot be proven within the comparison budget, CONSERVE is a formula error (fail loudly, never guess): a guard that cannot confirm its safe condition must not let flow pass, and a broken conservation assumption must never be degraded to a bubble that flows downstream. This strictness is the deliberate difference between CONSERVE (a safety assertion) and EQ (an observation that may return unknown). Over the admitted domain \(D\) (Section 4.2.7) sum-equality is exact and decidable, so for rational (money-scale) values CONSERVE either passes or fails loudly.

Table-invariant form (normative). A table may declare the invariant CONSERVE(total-cell, parts-range) as table metadata. The invariant is checked on every recalculation that dirties any participant. A violation marks the table with a machine-readable integrity failure — surfaced by the host as a loud, non-dismissable-by-formatting error state — while cell values remain whatever the formulas produced (the invariant observes; it does not rewrite values).

Relationship to QUANTIZE. QUANTIZE's residual identity \(q + r = x\) (Section 7.13) is the atomic conservation guarantee for a single rounding; CONSERVE is the aggregate one over a whole range of parts. Together they let a document quantize an allocation to a currency grid and then prove that the quantization neither invented nor lost value.

10.3 EXPECT: example-row assertions

A function table (Section 5.6) may designate one literal column as its EXPECT column. The designation declares a table invariant, checked under the same regime as CONSERVE's invariant form (Section 10.2): on every recalculation that dirties a participant, and fail loudly.

  • Assertion. For every example row whose EXPECT cell is non-Empty, the row's output-column value (Section 5.6) must equal the EXPECT value. Equality is value identity (Section 4.2.4 for Scalars); an Empty EXPECT cell asserts nothing for its row.
  • Fail loudly, never guess. A mismatch, an output value of NIL or U where EXPECT is definite, or an equality that cannot be proven within the comparison budget marks the table with a machine-readable integrity failure — the same loud, non-dismissable-by-formatting state as a CONSERVE violation. Cell values remain whatever the formulas produced; the invariant observes, it does not rewrite.
  • Role. Example rows with EXPECT are the function's unit tests, co-located with its definition and re-run by recalculation itself. An edit that breaks the function is visible at the moment it is made.

11.2 Bubble Rule

The boundary between the first two categories is the Bubble Rule:

  • Malformed use of a word is an error — the program itself is wrong, the cell has no value, and the fix is to the formula (recoverability = fixProgram).
  • A well-formed operation that misses its domain produces a Bubble — a NIL carrying the reason, flowing through NIL-passthrough words (Section 7.12) so that one ^ (VENT) at the pipeline's end can supply a fallback (recoverability = recoverable).

Errors are never values: an error state never flows into arithmetic. Bubbles are always values: they never abort recalculation. The grid displays both distinctly, with diagnosis on demand, and the semantics-driven display always wins over user formatting (Section 13.2).

13. Host Boundary and Presentation Layer

13.1 Effects are host commands

All effects — editing cells, defining words (DEF/DEL), structural edits, persistence, import/export, printing — are host commands issued through the GUI or the machine protocol, never words invocable from a formula (Section 6.3). The Script surface is the host's word-editing environment; it is where DEF is written, and it may offer imperative conveniences, but those conveniences are host features, not formula semantics.

13.2 Presentation is not value

Column widths, styles, cell merges, sheet layout, and formats live in the presentation layer, stored with the document but outside the value model. Normative constraints on presentation:

  • Semantics-driven display wins. The NIL / U / error / cycle display states cannot be painted over by user formatting.
  • Merges are display spans. A merge names an anchor cell; covered addresses resolve to the anchor for reference purposes; values are owned by the anchor only. Export of the logical model ignores merges.
  • Formats are quantization (Section 10.1) and never alter the exact value.
  • Truncated numeric display is marked. When a display shortens an exact value (long rationals, lazy CFs), the host must make the truncation perceivable and must never re-parse displayed text as the value (Sections 3.4, 4.2.3).

13.3 Persistence and interchange

The machine-readable document format serializes tables (cell formulas in canonical form, literals as exact values), function-table metadata (parameter declarations, output designation, EXPECT designation — Sections 5.6, 10.3), the dictionary, presentation metadata, and invariants. Exact values travel in a protocol form that round-trips losslessly; display strings are never the interchange form. CSV export writes display values and is lossy by declaration; CSV import reads literals under Section 3.6's entry convention.

14. AI-first Implementation Rules

14.1 Mandatory

  • The dependency graph, cell diagnostics, formula bodies (canonical form), contract metadata, and invariant states are exposed machine-readably at every point where a human can see their rendered equivalents.
  • Diagnostic protocol fields are lower camel case strings, stable across releases; human-readable summaries are non-canonical.
  • Every normative behavior in this specification is traceable to tests (Section 16); implementation modules cite the section they implement.
  • No implementation may branch on non-observable state (Section 2.3) to produce observable differences.

14.2 Advisory

  • Prefer structurally searchable implementation (one concept, one home) over clever fusion.
  • Keep the canonical formula form the stored form; the infix surface is derived at the edge.

  7. Implements QUANTIZE with the residual identity and banker's rounding (Section 7.13), format-as-quantization (Section 10.1), and CONSERVE with fail-loud semantics in both word and invariant forms (Section 10.2).
  8. Keeps the Bubble Rule boundary of Section 11.2, with machine-readable diagnoses on every error, bubble, U, and cycle outcome.
  9. Round-trips the canonical ⇄ infix surface bijectively for the grammar of Section 3.6.
  10. Implements function tables per Sections 5.6 and 6.6: positional parameter binding, default and named output columns, on-demand sibling-column evaluation, the memo soundness condition (memoization only on exactly decided argument equality, never observable in values), EXPECT invariants with fail-loud semantics (Section 10.3), and the call-recursion/reference-cycle distinction of Section 9.6.
  11. Exposes the machine-readable surfaces of Section 14.1.

Appendix A. Water on the Grid (non-normative index)

This appendix is non-normative. It introduces no new rules, types, words, or protocol fields. It is a vocabulary index that maps the water metaphor of Ajisai's user-facing material onto the normative mechanisms defined above, continuing the role of the previous canon's "Gates and Water Levels" appendix. Where this appendix and any numbered section disagree, the numbered section governs.

The metaphor carries over from the vector-oriented canon intact — and becomes more literal. A spreadsheet's own vocabulary already speaks of upstream and downstream cells: the dependency graph of Section 9.1 is a watershed, and recalculation is water finding its way downhill. The base image of the grid is the vessel (器): a cell is a vessel, and a vessel holds one water. Ajisai — the hydrangea — is the flower of the rainy season: many small vessels fed by one water system.

Water vocabularyNormative mechanism
Vessel (器) — one vessel, one waterThe cell (Sections 4.1, 6.2). The Flat Cell principle in a single image: a vessel holds water, never other vessels. There are no nested values; structure is the arrangement of vessels on the grid (Section 5), and a formula fills exactly one vessel with exactly one water (Section 6.2).
Watershed (流域) — upstream and downstreamThe dependency graph and its inverted index (Section 9.1). An edit is rainfall at a point; the dirty closure (Section 9.2) is exactly the watershed downstream of it, and topological recalculation (Section 9.3) is the water running downhill from vessel to vessel.
Confluence (合流) — parted streams rejoin as oneThe order- and parallelism-independence of recalculation (Section 9.3). The functional-programming term for this property is the water word: evaluation orders may branch and rejoin, and the settled document is the same.
Bubble (泡) — failed water rising through the flowReason-carrying NIL and its passthrough (Sections 4.5, 7.12, 11.2), received at one visible point by VENT ^ (Section 7.7). On the grid a bubble is visible in the cell it reaches.
Stagnation (淀み) — flow that neither passes nor failsThe logical Unknown (U) of an undecided observation (Sections 4.3, 7.4.1). A stagnant cell is visible as such; K3 logic (Section 7.5) settles what definite flow can settle and carries the stagnation otherwise.
Awaiting rain (雨待ち) — an empty vessel the rain has not yet reachedEmpty (Section 4.6). An edit is rainfall, and an untouched cell is an empty vessel: not failed water (a bubble carries its reason, Section 4.5), just none yet. Aggregation flows past empty vessels (Section 7.1), while treating an empty vessel as if it held water is refused (no blank-as-zero, Section 4.6).
Uphill channel (逆勾配の水路) — water does not run uphillA reference cycle (Section 9.4). Recalculation is water running downhill (Section 9.3); a cycle is a channel that could fill only if water flowed uphill, so it is refused when the flow is planned — diagnosed exactly, every member marked, never spun. Function-table call recursion is not uphill: each application steps down onto a fresh terrace, a cascade bounded by the vessel's water level (Section 9.6).
Water levels (水位) — how much flow may runPer-cell budgets (Sections 6.4, 9.5): each vessel has its own level, so one overflowing vessel never floods the document. Step-budget exhaustion is an operational outcome (NIL, reason = stepBudgetExhausted); an exhausted comparison budget is Stagnation (U, Section 7.4.1) — the previous canon's defining distinction between the two, preserved (Section 4.5.2).
Gates (水門) — where flow may cross a boundaryThe host boundary (Section 13.1). Outward: effects leave the pure grid only as host commands. Inward: values enter only as literal entry and import at the host surface (Sections 3.6, 13.3). Inside the gates, the grid is a closed, conserved water system (Section 6.3).
Conservation (保存) — no water invented or lostQUANTIZE's residual identity \(q + r = x\) captures the runoff of a single rounding instead of spilling it (Section 7.13); CONSERVE proves a whole allocation reconciles to its total (Section 10.2); EXPECT keeps a function table's example flows true (Section 10.3). All three fail loudly rather than let unproven flow pass.
